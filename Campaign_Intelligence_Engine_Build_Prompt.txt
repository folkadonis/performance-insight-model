================================================================================
SECTION 1: PRODUCT OVERVIEW
================================================================================

Build a multi-tenant, hierarchically scoped AI-Powered Campaign Performance
Intelligence Engine that ingests campaign data at 4 levels:

    Industry --> Market --> Tenant --> Business Unit (BU)

The system builds a rich ML context from trained models at each level, and feeds
that context into an LLM to generate structured, actionable campaign insights
inside the Campaign Summary Report (CSR).

SECTION 3: HIERARCHY MODEL
================================================================================

SCOPE RESOLUTION ORDER
----------------------
Each campaign is resolved through all 4 levels. The ML model and benchmarks
are pulled from the most specific scope available, falling back upward if
data is insufficient.

    BU Level           --> Primary scope (most specific)
        ^ fallback
    Tenant Level       --> Secondary scope
        ^ fallback
    Market Level       --> Tertiary scope
        ^ fallback
    Industry Level     --> Base scope (broadest benchmarks)

SECTION 4: ML MODEL LAYER
================================================================================

4.1 WHAT THE ML MODELS DO
--------------------------
Each scope level has a trained model. All models produce a scored feature
vector - not a prediction alone, but a rich set of scored signals about
the campaign.

ML Model Input:
  - Campaign metadata
  - Audience features (from CDP)
  - Channel performance metrics
  - Journey execution data
  - Historical campaign data (same scope)

ML Model Output (Feature Vector):
  - reach_score               --> 0.0-1.0
  - engagement_quality_score  --> 0.0-1.0
  - channel_efficiency_score  --> per channel
  - audience_fit_score        --> 0.0-1.0
  - timing_quality_score      --> 0.0-1.0
  - journey_effectiveness     --> 0.0-1.0
  - frequency_risk_score      --> 0.0-1.0
  - churn_signal_score        --> 0.0-1.0
  - cross_sell_opportunity    --> 0.0-1.0
  - conversion_probability    --> 0.0-1.0
  - benchmark_delta           --> % above/below peer average
  - anomaly_flags             --> list of detected anomalies


4.2 MODEL TYPES PER SCOPE
--------------------------
Scope      | Model Type                   | Trained On                        | Purpose
-----------|------------------------------|-----------------------------------|--------------------------------
Industry   | XGBoost / LightGBM           | All tenants in industry           | Industry-wide patterns/baselines
Market     | XGBoost + Regional Features  | All tenants in market             | Regional behavioral patterns
Tenant     | Fine-tuned Gradient Boosting | Tenant's own historical campaigns | Tenant-specific fingerprint
BU         | Fine-tuned + Product-aware   | BU's campaigns only               | Most precise product-level score


4.3 FEATURE CATEGORIES (150-500 features per campaign)
-------------------------------------------------------
Audience Features (40-80):
  segment_size, age_distribution, gender_mix, affluence_index,
  geography_code, device_preference, product_ownership_count,
  engagement_score, rfm_recency, rfm_frequency, rfm_monetary,
  propensity_score, clv, churn_probability, channel_affinity_email,
  channel_affinity_sms, channel_affinity_whatsapp, preferred_hour,
  previous_campaign_response_rate, days_since_last_purchase ...

Campaign Features (20-40):
  campaign_objective_encoded, campaign_type_encoded, offer_category,
  discount_pct, urgency_flag, promotion_type, journey_step_count,
  communication_frequency, execution_day_of_week, execution_hour,
  campaign_duration_days, industry_vertical, product_category ...

Channel Features (30-60, per channel):
  email_delivered_rate, email_open_rate, email_click_rate,
  email_conversion_rate, email_unsubscribe_rate, email_bounce_rate,
  email_spam_rate, email_revenue_contribution,
  sms_delivered_rate, sms_click_rate, sms_conversion_rate,
  whatsapp_open_rate, whatsapp_reply_rate, whatsapp_conversion_rate,
  push_open_rate, push_conversion_rate ...

Historical Features (20-40):
  avg_reach_last_10, avg_conversion_last_10,
  best_performing_channel_last_5, avg_open_rate_same_segment,
  avg_revenue_same_product, conversion_trend_slope,
  best_day_of_week_historically, best_hour_historically ...

Benchmark Delta Features (20-40):
  delta_vs_tenant_avg, delta_vs_bu_avg, delta_vs_market_avg,
  delta_vs_industry_avg, percentile_rank_in_tenant,
  percentile_rank_in_industry ...

SECTION 6: LLM PROMPT DESIGN
================================================================================

SYSTEM PROMPT
-------------
You are a trained Campaign Performance Intelligence Engine for [TENANT_NAME]
operating within the [INDUSTRY] industry, [MARKET] market, [BU] business unit.

You have been provided a structured ML-scored campaign context. Your job is to
analyze this context and generate precise, quantitative, actionable insights.

Hierarchy-aware rules:
- When scores fall below BU benchmarks, flag as BU-level underperformance
- When scores are above industry average but below BU average, note the gap
- Always reference the most specific benchmark level available
- Cross-sell and opportunity insights must reference BU product catalog

Output: Return ONLY a valid JSON array. No preamble. No markdown.
Each object must follow this exact schema:

{
  "insight_type"    : one of [Performance, Root Cause, Audience, Channel,
                              Journey, Timing, Opportunity, Anomaly,
                              Forecast, Prescriptive],
  "category"        : one of [Opportunity, Risk, Recommendation,
                              Benchmark, Prediction],
  "scope"           : one of [BU, Tenant, Market, Industry],
  "observation"     : "Factual statement of what happened with numbers",
  "root_cause"      : "ML-backed explanation of why it happened",
  "recommendation"  : "Specific, executable next action",
  "business_impact" : "Quantified expected uplift or risk value",
  "confidence"      : integer 50-99
}

Generate 8-12 insights. Cover all 10 insight types. Do not repeat types
more than twice. Be specific, quantitative, and hierarchy-aware.


USER PROMPT (assembled by Context Builder)
------------------------------------------
Analyze the following ML-scored campaign context and generate insights:

[PASTE FULL CONTEXT BLOCK HERE]

Prioritize:
1. Anomalies flagged by the ML model
2. Largest benchmark deltas
3. Highest cross-sell opportunity scores
4. Channels with efficiency score gaps > 0.3

SECTION 8: API AND SERVICE DESIGN

ENDPOINTS
---------

POST /api/v1/insights/generate
  Body: {
    campaign_id       : string,
    tenant_id         : string,
    bu_id             : string,
    market_id         : string,
    industry_id       : string,
    campaign_metrics  : { ... },
    campaign_metadata : { ... }
  }
  Response: {
    context_block : string,
    ml_scores     : { ... },
    insights      : [ InsightObject x 8-12 ]
  }

GET /api/v1/insights/{campaign_id}
  --> Returns cached insight set for CSR display

GET /api/v1/benchmarks/{scope}/{scope_id}
  --> Returns benchmark profile for a given scope level

GET /api/v1/ml/models/{tenant_id}/{bu_id}
  --> Returns active model version metadata


PROCESSING FLOW
---------------
Step 1  : Receive campaign_id + hierarchy IDs
Step 2  : Resolve hierarchy --> fetch benchmarks at all 4 levels
Step 3  : Pull CDP audience features for campaign segment
Step 4  : Fetch channel metrics from campaign execution logs
Step 5  : Load historical campaigns (last 10, same scope filters)
Step 6  : Run ML model at BU level --> get feature vector + scores
            - If BU model confidence < threshold --> fallback to Tenant
            - If Tenant model confidence < threshold --> fallback to Market
            - Always merge Industry-level benchmarks
Step 7  : Assemble context block (structured text)
Step 8  : Send system prompt + context block to LLM
Step 9  : Parse JSON response --> validate schema
Step 10 : Store insights against campaign_id
Step 11 : Return to CSR for display

-- Hierarchy tables

CREATE TABLE industries (
  id                 VARCHAR PRIMARY KEY,
  name               VARCHAR NOT NULL,
  benchmark_profile  JSONB
);

CREATE TABLE markets (
  id                   VARCHAR PRIMARY KEY,
  name                 VARCHAR NOT NULL,
  industry_id          VARCHAR REFERENCES industries(id),
  regional_profile     JSONB
);

CREATE TABLE tenants (
  id                VARCHAR PRIMARY KEY,
  name              VARCHAR NOT NULL,
  market_id         VARCHAR REFERENCES markets(id),
  ml_model_version  VARCHAR
);

CREATE TABLE business_units (
  id                VARCHAR PRIMARY KEY,
  name              VARCHAR NOT NULL,
  tenant_id         VARCHAR REFERENCES tenants(id),
  product_category  VARCHAR,
  ml_model_version  VARCHAR
);

-- ML model registry

CREATE TABLE ml_models (
  id                      UUID PRIMARY KEY,
  scope_level             VARCHAR,
  scope_id                VARCHAR,
  model_version           VARCHAR,
  model_path              VARCHAR,
  feature_count           INTEGER,
  trained_on_date         DATE,
  min_campaign_threshold  INTEGER,
  fallback_scope_id       VARCHAR
);

-- Campaign insights store

CREATE TABLE campaign_insights (
  id               UUID PRIMARY KEY,
  campaign_id      VARCHAR NOT NULL,
  tenant_id        VARCHAR,
  bu_id            VARCHAR,
  context_block    TEXT,
  ml_scores        JSONB,
  insights         JSONB,
  model_version    VARCHAR,
  llm_model        VARCHAR,
  generated_at     TIMESTAMP DEFAULT NOW(),
  confidence_avg   FLOAT
);

-- Benchmark profiles

CREATE TABLE benchmarks (
  id           UUID PRIMARY KEY,
  scope_level  VARCHAR,
  scope_id     VARCHAR,
  metric_name  VARCHAR,
  avg_value    FLOAT,
  p25          FLOAT,
  p50          FLOAT,
  p75          FLOAT,
  p90          FLOAT,
  period       VARCHAR,
  updated_at   TIMESTAMP
);

-- Insight deduplication log

CREATE TABLE insight_history (
  id                UUID PRIMARY KEY,
  bu_id             VARCHAR,
  campaign_id       VARCHAR,
  insight_type      VARCHAR,
  observation_hash  VARCHAR,
  generated_at      TIMESTAMP
);

SECTION 12: KEY DESIGN PRINCIPLES
================================================================================

1. HIERARCHY-FIRST SCORING
   Every score and insight is tagged to its scope level so users always know
   whether a benchmark is BU-specific or industry-wide.

2. ML SCORES DRIVE LLM PROMPTS
   The LLM never sees raw metrics alone. It receives ML-scored signals so
   insights are model-backed, not pattern-matched from text.

3. GRACEFUL FALLBACK
   If BU data is insufficient, the system falls back:
   BU --> Tenant --> Market --> Industry
   Fallback scope is always flagged in the insight output.

4. INSIGHT DEDUPLICATION
   System checks the last 3 CSRs for the same BU and suppresses verbatim
   repeated insight types, forcing new observations each run.

5. CONFIDENCE GATING
   Insights with model confidence below 55% are suppressed or flagged
   as low-confidence to avoid misleading users.

6. CONTEXT RICHNESS OVER RAW DATA
   Raw numbers are never sent to the LLM alone. The context block includes
   ML scores, benchmark deltas, anomaly flags, and historical context to
   ground the LLM for specific, non-generic insights.

7. SCOPE-AWARE RECOMMENDATIONS
   Cross-sell and opportunity insights always reference the specific BU
   product catalog so recommendations are relevant to what the tenant sells.


================================================================================
SECTION 13: INSIGHT QUALITY GUARDRAILS (LLM RULES)
================================================================================

The LLM system prompt must enforce:

- Every observation must contain at least one number or percentage
- Root cause must reference an ML score or a benchmark delta
- Recommendation must be a specific executable action (not generic advice)
- Business impact must be quantified (revenue, percentage uplift, or count)
- Confidence score must reflect ML model certainty, not LLM confidence
- No two insights of the same type should have similar observations
- All scope references (BU/Tenant/Market/Industry) must be accurate
- Anomaly insights only generated if anomaly_flags list is non-empty
- Forecast insights must reference trend slope or day-of-campaign data
- Prescriptive insights must cite which ML score triggered the recommendation
